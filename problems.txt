# Engine-level limitations (scenario generation)

## 1. Entity-level hiding not supported
   The engine surfaces every entity in `entities_present` to the LLM
   unconditionally — there's no `hidden` flag on entities (only on
   exits). The spider in the webs and the handkerchief in the rubbish
   currently rely on LLM compliance. A `hidden: true` field on entity
   definitions, or a `revealed` state field that the GMBriefing filters
   on, would be needed for engine-enforced entity visibility.

## 2. No global traversal check mechanism
   If carrying an item (like the heavy key) affects movement, the
   `traversal_check` must be duplicated on every exit. There's no
   engine-level way to say "when item X is in inventory, apply check
   Y to all traversal." A `global_rules` block or a
   `modifies_movement` property on items could address this.

## 3. No dynamic DC scaling on stat checks
   Escalating DCs (e.g., Korbar's CHA check: DC 9 → 12 → 15 per
   attempt) require separate interaction entries with mutually
   exclusive conditions. This is verbose and error-prone. A
   `dc_increment` or `dc_escalation` field on StatCheck would let
   the engine handle it natively.

## 4. Game-over from Result is always indirect
   There's no way for a Result (or chain_check) to trigger game-over
   directly. The `set_flag` → `lose` mechanic pattern works but
   requires two-step setup. May be intentional (engine philosophy:
   game-over is exclusively mechanic-evaluated).

## 5. NPC blocking only covers predefined interactions
   The NPC action blocking pattern (e.g., Korbar shooing the player)
   only works for corpus-defined interactions in the room. Generic
   actions like `examine room` can't be blocked by NPC attitude —
   those rely on LLM compliance.

# Other design Issues

## Consider pruning LLM instructions situationally
For example, if there's no NPC present, we can omit the complicated
NPC-handling instructions.

## STALE ROOM DESCRIPTIONS PROBLEM
    The axe_handle_lower description mentions the spider; after it
    flees, the description is factually incorrect.  The briefing
    includes entity state alongside the description, so the LLM can
    mentally override, but this is fragile.  This is a schema-level
    limitation that won't be fixed in Phase 2 but should be kept in
    mind when designing the Context Assembler — it should prominently
    flag entity state next to stale descriptions.

    This is a content/design concern for the Context Assembler (Phase
    5), not a model defect.  The Entity model and BriefingEntity
    already include state fields; the assembler should present them
    prominently alongside descriptions.

## Encounter resolution happens *after* hard-state changes are committed
    Per `phase-4-plan.md` §Step 4, encounters must fire *before* hard-state changes so that a death outcome does not leave the player in a contradictory state (e.g. moved to a new room and died simultaneously).  Current ordering: `apply_hard_changes()` → `resolve_encounter()`.  If the encounter produces `game_over`, the player location has already been updated.  
**Fix:** Reorder to evaluate encounters against the *proposed* changes before committing them; only commit hard changes if the encounter does not produce `game_over` (or commit them conditionally).

## `resolve_examine` completely ignores the `rigorous` flag
    File: `mgmai/engine/resolver.py` (`resolve_examine`)  
    The `rigorous` field on `ExamineAction` is never read.  The schema requires that a rigorous examination evaluate interactions that are gated behind `rigorous` searches. Fix: if `action.rigorous` is `True`, evaluate the target entity/room for any interactions whose availability depends on `rigorous` and include the results in `triggered_narration`.
